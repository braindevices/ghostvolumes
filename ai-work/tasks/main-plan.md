# GhostVolumes ŌĆö Implementation Plan

**Goal:** Automatically isolate volatile build artifacts (`node_modules`, `target`, `.venv`, `build`, etc.) into unsnapshotted BTRFS subvolumes, with zero sudo at runtime and effectively zero overhead for the common case.

---

## 1. Scope for this phase

- **Interception mechanism:** LD_PRELOAD only. Seccomp/ptrace-based syscall interception (needed to catch statically-linked binaries, e.g. the musl-static `uv` build used in its official Docker images) is **shelved** until observed as a real problem in practice.
- **Snapshot-manager detection:** no runtime probing. Fully resolved ahead of time by `ghostvolumes scan --save` (or hand-edited) into a static config that the runtime hot path just reads.
- **Git-tracked content:** never touched ŌĆö neither proactively created nor converted. This is a scope boundary, not a workaround: GhostVolumes isolates *disposable* artifacts, and anything git-tracked is by definition not disposable. (Note: `rmdir()` has been able to remove populated/empty BTRFS subvolumes like normal directories since kernel 4.18, so this exclusion is about intent, not a filesystem limitation.)

---

## 2. Config layout ŌĆö drop-in style

Defaults to XDG user paths (`~/.config/ghostvolumes/`, `~/.local/share/ghostvolumes/`) rather than `/etc`, since `cargo install` is inherently a per-user mechanism with no root step and no postinstall hook. `/etc/ghostvolumes/` is supported as an optional lower-precedence tier for shared multi-user machines (loaded first, user config layers on top with the same merge rules), but nothing in the default flow requires it or requires sudo to set up.

```
~/.config/ghostvolumes/               # default; /etc/ghostvolumes/ optional, lower-precedence
Ōö£ŌöĆŌöĆ roots.d/
Ōöé   Ōö£ŌöĆŌöĆ 00-auto.toml                  # written by `ghostvolumes scan --save` ŌĆö do not hand-edit
Ōöé   ŌööŌöĆŌöĆ 10-local.toml                 # hand-edited additions (e.g. Timeshift/btrbk roots)
Ōö£ŌöĆŌöĆ watched.d/
Ōöé   Ōö£ŌöĆŌöĆ 00-defaults.toml              # ships with package: node_modules, target, .venv, build
Ōöé   ŌööŌöĆŌöĆ 10-local.toml                 # hand-edited global additions
ŌööŌöĆŌöĆ projects.d/
    ŌööŌöĆŌöĆ local.toml                    # explicit per-project opt-in (see ┬¦4)
```

Plus an optional **repo-local** `.ghostvolumes.toml`, auto-discovered by the cd-hook (`ensure`) walking up from `$PWD` and folded into the compiled cache the runtime paths actually read ŌĆö see ┬¦6 ŌĆö so project-specific config can be checked into the repo itself.

- Files in a `*.d/` dir load in lexical order; `00-` = generated, `10-` = hand-edited, by convention.
- `roots.d` / `watched.d` are simple sets ŌĆö union + dedupe across files, no precedence conflicts.
- `.ghostvolumes.toml` layers in last, closest to the code being run, once registered by the cd-hook (┬¦6) ŌĆö see that section for what happens before the first registration.
- `scan --save` only ever writes `roots.d/00-auto.toml`. Never touches anything else.

### Example: `roots.d/00-auto.toml` (generated)
```toml
# GENERATED by `ghostvolumes scan --save`. Re-run to refresh.
# Manual additions belong in roots.d/10-local.toml instead.
roots = ["/", "/home", "/dbs"]
```

### Example: `roots.d/10-local.toml` (hand-edited)
```toml
# Snapshot-managed mountpoints scan couldn't auto-detect (e.g. Timeshift, btrbk).
roots = [
    # "/srv/www",
]
```

### Example: `watched.d/00-defaults.toml`
```toml
names = ["node_modules", "target", ".venv", "build"]
```

### Example: `projects.d/local.toml`
```toml
[[project]]
path      = "/home/user1/projects/big-frontend"
watch     = ["dist", ".next"]     # additional reactive-only names, beyond global defaults
proactive = ["node_modules"]      # also automatically included in the reactive set

[[project]]
path      = "/home/user1/projects/rust-app"
proactive = ["target"]
```

## 3. Snapshot-root detection (`ghostvolumes scan`)

Layered by privilege, resolved once and cached ŌĆö never on the runtime hot path.

1. **Unprivileged, always run:** walk up from each BTRFS mountpoint checking for a `.snapshots` sibling that is itself a subvolume (`stat().st_ino == 256`). This is Snapper's structural fingerprint ŌĆö always present for any Snapper config, hardcoded name, and visible via plain `stat()` regardless of whether it's separately mounted (confirmed: nested subvolumes are fully traversable through their parent without a dedicated mount entry). Correctly handles both openSUSE-style (separately mounted `.snapshots`) and Arch-style (nested, unmounted) layouts.
2. **Privileged, only if run under `sudo`:** query `snapper list-configs`, parse `/etc/timeshift/timeshift.json`, parse `/etc/btrbk/btrbk.conf`. Needed because Timeshift's marker lives outside any subvolume an unprivileged user can see, and btrbk's snapshot directory name is arbitrary config, not a fixed convention.
3. Manual entries in `roots.d/10-local.toml` always supplement auto-detected ones.

**Command:**
```
ghostvolumes scan            # dry run, prints resolved list
ghostvolumes scan --save     # atomically writes roots.d/00-auto.toml
```

## 4. Path-matching logic (shared by both runtime code paths)

```
resolve_watch_names(path, roots, global_defaults, projects):   # used by LD_PRELOAD ŌĆö reactive
    if path is not under any entry in roots:
        return Ōłģ                  # cheapest possible rejection, checked first
    if no [[project]] entry matches an ancestor of path:
        return global_defaults
    return global_defaults Ōł¬ project.watch Ōł¬ project.proactive

resolve_proactive_names(path, roots, projects):                 # used by cd-hook ŌĆö proactive
    if path is not under any entry in roots:
        return Ōłģ
    if no [[project]] entry matches an ancestor of path:
        return Ōłģ                  # no project match => cd-hook does nothing
    return project.proactive minus any name failing is_git_tracked()
```

The root check is the first thing both functions do, by construction ŌĆö never optional, never left to a caller to remember ŌĆö because it's the cheapest possible rejection and the plan's own zero-overhead goal (┬¦Goal) depends on the overwhelming majority of intercepted calls (everything outside GhostVolumes-managed territory) being rejected this cheaply. In the compiled `compiled.tsv` reader (┬¦8.0) this check isn't a separate step at all: because every row's prefix is already root-scoped, one pass over the rows checking "is this prefix an ancestor of path" does root-rejection and name-matching simultaneously ŌĆö the two are the same operation there, not sequential ones.

Key rules baked into this:
- **Reactive (LD_PRELOAD) always applies globally** under any configured root, using at least the global default names ŌĆö this is the safety net that catches new/unexpected paths (e.g. a nested `packages/foo/node_modules` in a monorepo) even without a project entry.
- **Proactive (cd-hook) requires an explicit per-project opt-in.** No project match ŌåÆ no pre-creation. This avoids blindly creating four empty subvolumes in every directory the user happens to `cd` into.
- `proactive` names are automatically part of the reactive set ŌĆö no need to list a name in both `watch` and `proactive`.

### Git-tracked gate
```
is_git_tracked(path):
    repo_root = walk up from path looking for .git (tolerate "not a repo")
    if no repo found or git binary missing: return false
    return `git -C repo_root ls-files -- <path>` produces output
```
Applied at all three call sites:
- **cd-hook:** skip that name for that project, one warning per repo/session (not per `cd`).
- **LD_PRELOAD:** pass through to the real `mkdir` untouched.
- **`convert`:** hard refusal, no override ŌĆö see ┬¦7.

Cheap to check: only runs after the name/prefix filter already matched (a handful of times per build), never on every syscall.

## 5. LD_PRELOAD hook (reactive, runtime)

1. `constructor` function reads the compiled `compiled.tsv` cache (see ┬¦8.0) once at process start into an in-memory `(prefix, name)` list ŌĆö no TOML parsing happens in the shim itself.
2. Intercepts **both** `mkdir` and `mkdirat` ŌĆö confirmed via testing that real tools use either (glibc `uv` build used only `mkdir`; other tools/libc versions commonly use `mkdirat`).
3. Resolves the target to an absolute path: a relative `mkdir` path is prefixed with `getcwd()`; a relative `mkdirat` path against a non-`AT_FDCWD` dirfd is resolved via `readlink("/proc/self/fd/<dirfd>")` ŌĆö the only portable way to recover a path from a bare fd without extra crates ŌĆö then joined. Real-world traces show a mix of absolute and relative arguments even within a single tool invocation, so both must be handled.
4. **Root-prefix check first, before any name matching.** LD_PRELOAD intercepts every `mkdir`/`mkdirat` call system-wide, so the hot path must reject the overwhelming majority of calls (anything outside GhostVolumes-managed territory) as cheaply as possible: a handful of string-prefix comparisons against the `roots` list. Every row in `compiled.tsv` is already rooted under one of these prefixes (┬¦8.0), so in practice this check and the name lookup below can be one pass over the compiled rows ŌĆö but conceptually root-rejection is the fastest, first-line filter, and nothing more expensive happens for a path outside every configured root.
5. Checks resolved path against `resolve_watch_names()` for the parent directory (only reached once the root check above passes).
6. If matched, not already a subvolume (`st_ino != 256`), and not git-tracked ŌåÆ issue `BTRFS_IOC_SUBVOL_CREATE` instead of the real call. This order is deliberate: `stat()` (cheap-ish syscall) before shelling out to `git` (much more expensive) ŌĆö never do the expensive check before the cheap one already ruled things out.
7. Tolerates `EEXIST` gracefully ŌĆö real traces show tools retry directory creation bottom-up after an initial `ENOENT` on the leaf, which looks like duplicate `mkdir` calls for the same path.

**Validated in testing:** a glibc-dynamically-linked `uv venv` run showed the hook correctly intercepting every `mkdir` call for `.venv`, `bin/`, `lib/python3.14/site-packages`, etc., including the duplicate-call and mixed absolute/relative path patterns described above.

**Known gap (accepted for now):** statically-linked binaries (e.g. `uv`'s musl-static Docker build) issue raw syscalls that bypass libc entirely ŌĆö LD_PRELOAD cannot intercept these. Revisit with seccomp-notify only if this is observed to matter in practice.

## 6. cd-hook (proactive, runtime)

- Shipped as a thin shell snippet (bash `cd` wrapper / zsh `chpwd`) that shells out to a fast binary: `ghostvolumes ensure "$PWD"`. All matching logic lives in the shared binary, not duplicated in shell.
- `ensure` reads `compiled.tsv` (┬¦8.0) for its fast prefix check ŌĆö the same flat cache the shim reads. It does not re-parse/re-merge the `*.d/` TOML on every `cd`; that would defeat the "must stay cheap" goal below.
- On entering a directory: calls `resolve_proactive_names($PWD)`.
  - No project match ŌåÆ **no-op immediately**, no filesystem work at all.
  - Project match, name missing ŌåÆ create it now as an empty subvolume, ahead of any tool run. This is what neutralizes most of the static-binary gap in practice ŌĆö by the time a tool runs, the directory already exists.
  - Project match, exists as plain directory ŌåÆ leave alone, warn once per session (likely pre-existing data ŌĆö see `convert`). "Session" means one interactive shell's lifetime: since `ensure` is a fresh process every invocation with no memory of prior ones, the dedup key is the shell's own `$$` (already available, no new env var needed) plus a hash of the project path. `ensure` touches a marker at `${XDG_RUNTIME_DIR:-/tmp}/ghostvolumes/warned/$$-<project-hash>` and only warns if it's not already there ŌĆö `XDG_RUNTIME_DIR` is normally tmpfs and cleared on logout, so no explicit cleanup is needed.
  - Project match, exists as subvolume already ŌåÆ no-op.
- **Repo-local `.ghostvolumes.toml` registration:** on a project-root match, `ensure` additionally checks (one cheap `stat()`) whether that root has a `.ghostvolumes.toml` newer than what's already reflected in `compiled.tsv`. If so, it parses that one file ŌĆö `ensure` runs as the full CLI, so it has the same TOML access `scan`/`discover` do ŌĆö and triggers the same reload/regenerate step ┬¦8.0 already runs after `scan --save`/`discover --save`, folding this repo's entries in. This is what makes repo-local config a real peer of the global config despite the shim itself never parsing TOML: it turns "this file doesn't exist as far as the reactive path is concerned" into "covered starting from the first interactive `cd` into this repo." Before that first registration ŌĆö e.g. a non-interactive shell that clones and immediately builds without ever `cd`-ing interactively ŌĆö only the global default names apply reactively, which is the same already-documented non-interactive-shell gap below, not a new one.
- Must stay cheap: fast prefix check first, bail immediately if `$PWD` isn't under any configured root.

**Known gaps (documented, not solved here):**
- Non-interactive shells that don't source rc files (CI runners, `bash -c`, some IDE terminals) don't get pre-emptive creation ŌĆö they rely on LD_PRELOAD alone, which needs to be set globally (e.g. `/etc/environment`) to reach those contexts.
- Mid-session `rm -rf node_modules && npm install` recreates the directory after the `cd` already ran ŌĆö falls back to LD_PRELOAD catching the recreation.

## 7. Supporting subcommands

### `ghostvolumes discover [PATH] [--max-depth N] [--save]`
Bootstraps `projects.d` entries from subvolumes that already exist (pre-adoption or manual `btrfs subvolume create` usage).
- Stat-walks from `PATH` (default `$HOME`), skips `.git` and descending into matches.
- For each directory entry matching a watched name, checks `st_ino == 256`.
- Groups matches by parent directory, checks each against `is_git_tracked` and annotates skipped suggestions rather than silently omitting them.
- **Default: print-only**, formatted as ready-to-paste TOML. `--save` (explicit opt-in) appends to `projects.d/local.toml`.

### `ghostvolumes convert <path>`
One-time migration for a pre-existing, populated plain directory:
1. Refuse outright if `is_git_tracked(path)` ŌĆö no override.
2. Create a new subvolume at a temp path.
3. `cp -a --reflink=always` existing contents in (cheap on BTRFS ŌĆö extent-sharing metadata, not a real copy ŌĆö but still a full tree walk, so cost scales with file count not size).
4. Atomically swap into place, remove the old plain directory.

## 8. Packaging & distribution via `cargo install`

`cargo install` only installs `[[bin]]` targets ŌĆö it does not install or place `cdylib`/`.so` artifacts anywhere, and there's no postinstall hook (unlike apt/Homebrew). Two consequences shape the packaging design, plus a related one about config parsing:

### 8.0 The shim can't parse TOML ŌĆö compile config down to a flat cache instead

Bare `rustc` has no dependency resolution, so the shim can't pull in `toml`/`serde` (or their transitive dependency chains) the way the main CLI can. Rather than hand-vendoring a parser chain, split the responsibility cleanly: **only the CLI binary ŌĆö a normal `cargo`-built target with full crate access ŌĆö ever parses TOML.** It resolves and merges `roots.d/*` + `watched.d/*` + `projects.d/*` per ┬¦2 and ┬¦4, then compiles the result down into a flat, trivially-parseable cache the shim reads at each process startup:

```
# ~/.local/share/ghostvolumes/compiled.tsv ŌĆö machine-generated by `ghostvolumes reload`, do not edit
/home/user1	node_modules
/home/user1	target
/home/user1	.venv
/home/user1	build
/data/workspaces	node_modules
/data/workspaces	target
/data/workspaces	.venv
/data/workspaces	build
/home/user1/projects/big-frontend	dist
/home/user1/projects/big-frontend	.next
/home/user1/projects/big-frontend	node_modules	proactive
```

Tab-separated `(prefix, name)` pairs, with an **optional third column** marking a row `proactive`. **The global-default rows are keyed by each entry in `roots` (┬¦3), not a hardcoded `/`** ŌĆö `roots = ["/home/user1", "/data/workspaces"]` above produces one row per `(root, global default name)` pair. This is the mechanism that makes `roots.d` a real, cheap, first-line filter rather than a paper requirement: a row's prefix is never broader than an actual configured root, so a path outside every configured root simply has no matching row at all, and the shim rejects it with the same prefix-scan it was already going to do ŌĆö no separate root-list lookup needed. (A single-root Snapper setup with `roots = ["/"]` degenerates to the earlier `/`-prefixed form, which is still correct, just a special case rather than the default assumption.) Project rows list only that project's own `watch Ōł¬ proactive` names, not the global defaults again ŌĆö a reader matches a target path against every row whose prefix is an ancestor and unions the names, so a project row's names always apply on top of whichever root row also matches.

**The third column is what lets `ensure` (┬¦6) recover just the proactive subset.** The shim's reactive matching (`resolve_watch_names`) needs the full `watch Ōł¬ proactive` union and doesn't care which names are proactive ŌĆö it ignores the third column entirely. But the cd-hook's `resolve_proactive_names` needs *only* `project.proactive`, never `watch`-only names (┬¦4: "no need to blindly create four empty subvolumes in every directory"), and `ensure` can't recover that distinction from an undifferentiated `watch Ōł¬ proactive` union ŌĆö a row has to carry it explicitly. `parse()` returns `(prefix, name, is_proactive)` triples; `names_for()` (shim's reactive matching) ignores the third field, `proactive_names_for()` (ensure's proactive matching) filters on it. Global-default rows (root-keyed) and plain `watch` rows never carry the marker; only names actually in a project's `proactive` list do, whether or not that name is *also* in `watch` for the same project.

The shim's constructor just reads this file and `str::split('\t')`s each line into an in-memory `Vec<(String, String, bool)>` ŌĆö no external crate needed, and meaningfully cheaper per process launch than a real TOML parse would have been, which fits the zero-overhead goal better anyway.

**Root validation happens at compile time, not on the hot path.** Checking "is this actually BTRFS" on every intercepted `mkdir` would add syscall cost to a path that needs to stay cheap. Instead, `reload` (and `scan --save`/`discover --save`, which both end by calling it) validates every entry in the merged `roots` list is still a BTRFS-backed mountpoint (via `statfs` ŌĆö checking the filesystem magic number, not walking `/proc/self/mountinfo`) before compiling `compiled.tsv`, and **fails loudly** if one isn't: a root that used to be BTRFS and no longer is (repartitioned, unmounted, config hand-edited wrong) means the config is stale, and the fix is a fresh `scan --save` or a manual `roots.d` edit ŌĆö not silently compiling a cache that references a filesystem that can't actually take a `BTRFS_IOC_SUBVOL_CREATE`.

**Regeneration** (`systemctl daemon-reload`-style):
- Automatic at the end of `scan --save` and `discover --save`.
- Explicit `ghostvolumes reload` after any hand-edit to a `*.d/` file.
- Automatic the first time `ensure` (cd-hook) encounters a project-root `.ghostvolumes.toml` newer than what's already folded in ŌĆö see ┬¦6.
- Optional staleness guard: the shim can cheaply `stat()` the cache file's mtime against the newest source `*.toml` mtime and print a one-time stderr nudge to run `ghostvolumes reload` if stale ŌĆö still just a timestamp comparison, no parsing.

### 8.1 Compile the shim once at build time via `build.rs`; `init` just extracts it ŌĆö no runtime `rustc` dependency

Building the shim as a separate `cdylib` crate and hoping `cargo install` distributes it doesn't work ŌĆö the artifact would have nowhere to go. Cargo's "artifact dependencies" feature (`-Z bindeps`) exists specifically for embedding a `cdylib` built from a sibling crate ŌĆö its own motivating example is literally an LD_PRELOAD profiling shim ŌĆö but it remains nightly-only with no stabilization date, so it's not usable for a normal `cargo install` on stable Rust.

Keep the shim's entry point (`shim/preload.rs`) as a small, **crate-dependency-free** file: no `extern crate`/`use` of anything from crates.io. "Dependency-free" specifically means no crates.io crates ŌĆö it does *not* mean no `std`. Bare `rustc` links `std` automatically regardless of Cargo, since `std` ships pre-compiled with the toolchain rather than being resolved from a registry; the constraint that actually forces hand-declared `extern "C"` blocks is that bare `rustc` (invoked directly, bypassing `cargo build`) has no Cargo.lock and thus no way to locate/link a crates.io crate's compiled `.rlib` ŌĆö a constraint `std` is simply exempt from. So the shim uses `std::process::Command`, `std::fs`, `std::env::current_dir()`, etc. freely for anything with a normal std equivalent, and hand-declares `extern "C"` only for the handful of things `std` has no wrapper for at all: `dlsym` (for `RTLD_NEXT` symbol resolution), `ioctl` (for `BTRFS_IOC_SUBVOL_CREATE`), and the exported `mkdir`/`mkdirat` replacement symbols themselves (which the dynamic linker resolves calls to ŌĆö these have to be raw C-ABI exports by definition, not something `std` could provide).

**Shared logic lives in dependency-free files, included by both sides ŌĆö not duplicated.** Several pieces of logic are needed by both the main CLI and the shim: `compiled.tsv` parsing/matching (┬¦8.0), the git-tracked gate (┬¦4), and the BTRFS subvolume primitives (inode-256 check, `BTRFS_IOC_SUBVOL_CREATE`). Where that logic has zero crate dependencies already (as written, `is_git_tracked` and the flat-cache `parse`/`names_for` functions only ever touched plain `std` types), it lives in a file under `shim/` and gets pulled into the main crate's normal module tree via `include!("../shim/whatever.rs")` ŌĆö a textual splice, not a separate compilation ŌĆö so both sides compile the literal same source rather than two independently-written implementations that only an equivalence test proves match. Only the BTRFS ioctl wrapper needs adjustment: the main-crate version returns `anyhow::Result` for its own error-handling convenience, but the shared core (as included by the shim) is written against plain `std::io::Result`, with the main crate's `anyhow`-based call sites absorbing that via `?`'s automatic `From` conversion. `#[cfg(test)]` module blocks inside a shared file are compiled by `cargo test` (main crate) but automatically skipped by the shim's plain `rustc` invocation (which never passes `--cfg test`), so test code can stay right alongside the shared logic in one file with no extra separation needed. This is a plain `mod`/`include!` mechanism ŌĆö a rustc-native, filesystem-based feature that needs no dependency resolution, unlike `extern crate`.

**Compile it at build time, not at runtime `init`.** `cargo install` already does a full local compile of `ghostvolumes` on the machine it'll run on ŌĆö a `rustc` toolchain is by definition present at that moment, since it's what's compiling `ghostvolumes` itself. So `build.rs`:
1. Shells out to `rustc --edition 2021 --crate-type cdylib -O -C panic=abort --target $TARGET -o $OUT_DIR/preload.so <manifest-dir>/shim/preload.rs` (`$TARGET` read from the `TARGET` env var Cargo sets for build scripts, so the shim always matches whatever triple the main binary is being built for ŌĆö a no-op passthrough for the ordinary same-machine `cargo install` case, but keeps things correct if this crate is ever cross-compiled). No copying into `$OUT_DIR` first ŌĆö `rustc` reads `shim/preload.rs` directly from the source tree (via `CARGO_MANIFEST_DIR`), and its `mod cache_core;`/`mod git_core;`/`mod btrfs_core;` declarations resolve to the sibling files in `shim/` the same way they would under `cargo build` ŌĆö module resolution is a compiler feature, not a Cargo one. If `$TARGET` contains `musl`, additionally pass `-C target-feature=-crt-static` ŌĆö see libc-matching requirement below.
2. `main.rs` embeds the result as a byte blob: `include_bytes!(concat!(env!("OUT_DIR"), "/preload.so"))`.

**Not compatible with `cargo binstall`.** A prebuilt-binary installer like `cargo binstall` downloads a binary built once by the maintainer's CI and never runs `build.rs` on the end user's machine at all ŌĆö meaning the embedded shim bytes would be compiled against the CI machine's libc, not the end user's, silently reintroducing the exact libc-family/version-matching problem the "build on the target machine via `cargo install`" design exists to avoid for free. GhostVolumes should not publish `cargo binstall`-compatible release artifacts; `cargo install` (build from source) is the only supported installation path for this reason, not merely a default preference.

`ghostvolumes init` (run once after install) then does no compilation at all ŌĆö it just writes those embedded bytes out to `~/.local/share/ghostvolumes/preload.so`:
1. Extract the embedded `.so` bytes to disk.
2. `-C panic=abort` is deliberate: the shim runs injected into arbitrary foreign processes, so any internal panic should hit `SIGABRT` immediately rather than unwind through a host call stack the Rust unwinder has no business touching.
3. The shim's one-time config load uses a hand-written `#[used] #[link_section = ".init_array"]` static function pointer as its constructor ŌĆö the same mechanism `__attribute__((constructor))` uses in C ŌĆö since crates like `ctor` aren't available without dependency resolution either.
4. Writes default config skeletons into `~/.config/ghostvolumes/` if not already present.
5. Idempotent ŌĆö safe to re-run after an upgrade: the new `ghostvolumes` binary carries a newly build-time-compiled `.so` blob (from that version's build.rs run), so re-running `init` just re-extracts the current version's bytes. Still no `rustc` invocation at this point, ever.

This means `rustc` is only ever needed once ŌĆö whenever the crate itself is compiled ŌĆö never at `init` time. Today that's always `cargo install` on the end user's own machine, so this is moot in practice.

**Libc matching is a correctness requirement for the shim specifically, not a portability nice-to-have.** LD_PRELOAD injection shares the host process's address space, including its already-loaded libc ŌĆö this is categorically different from the main CLI binary, which runs standalone and can be statically linked however is convenient for distribution. If the shim's libc doesn't match the host process's libc family, there's no clean "version mismatch" failure; instead two competing allocator implementations end up in one process (e.g. memory `malloc`'d by the host's glibc handed to the shim's musl `free`, or vice versa) ŌĆö real heap corruption, not a hypothetical. Concretely:
- **Never statically link the shim**, even when the main `ghostvolumes` binary is built fully static (e.g. a musl build for portability) ŌĆö static linking and LD_PRELOAD are a contradiction in terms, since the whole point is loading *into* another process's existing libc. This applies only to the shim; the main CLI's own linking choice is unaffected.
- **glibc host ŌåÆ glibc-dynamic shim.** For a shim distributed as a prebuilt binary across distros, it needs to be built against an old-enough glibc that its symbol-version requirements are satisfied everywhere it's meant to run (glibc is backward- but not forward-compatible) ŌĆö same "build old, run on anything newer" rule any prebuilt Linux binary needs, just more consequential here because of the shared-address-space failure mode above.
- **musl host (e.g. Alpine) ŌåÆ musl-dynamic shim**, which needs explicit handling: musl targets default to fully static linking (`+crt-static`), and older Rust toolchains refuse outright to build a `cdylib` under that default (`cannot compile ... because target ... does not support the cdylib crate type`) ŌĆö the compiler is directly confirming static-vs-shared-library is a real conflict. `build.rs` must pass `-C target-feature=-crt-static` for musl targets (already reflected in the `rustc` invocation above) to force dynamic linking. Even then, the resulting `.so` also picks up a dynamic dependency on `libgcc_s`, which Alpine's Docker image doesn't ship by default ŌĆö worth a README note that a musl host needs `libgcc` installed (`apk add libgcc`) for the shim to load.

For `cargo install`, both axes resolve for free: `build.rs` compiles the shim right there, against whatever libc that exact machine actually has, matching family and version automatically. It's only a *future* prebuilt-binary distribution channel (GitHub releases, distro packages ŌĆö not in scope for this phase, see ┬¦8 intro) that would need to handle this explicitly per released target triple in CI ŌĆö glibc targets built on an old-glibc baseline image (the same requirement the main CLI already has for portability, nothing shim-specific), plus a musl target/variant built with the `-crt-static` override and the `libgcc` note above.

Note none of this changes the coverage picture from ┬¦5 ŌĆö a preloaded shared library is still bound by the same dynamic-linker mechanism regardless of implementation language, so statically-linked *host* binaries (the actual build tool being run, e.g. a musl-static `uv`) remain an accepted gap either way.

### 8.2 Shell integration via `shell-init`, not rc-file editing

`cargo install` can't append anything to `.bashrc`/`.zshrc` for you, and silently rewriting a user's rc file is the wrong default regardless. Follow the same pattern as `starship`, `zoxide`, and `direnv` ŌĆö a subcommand that prints a shell snippet for the user to `eval`:

```bash
# ~/.bashrc / ~/.zshrc
eval "$(ghostvolumes shell-init bash)"   # or: zsh
```

`ghostvolumes shell-init <shell>` prints, to stdout:
- The `export LD_PRELOAD=...` (or an append if `LD_PRELOAD` is already set to something else) pointing at `~/.local/share/ghostvolumes/preload.so`.
- The `cd`-wrapper function (bash) or `chpwd` hook (zsh) calling `ghostvolumes ensure "$PWD"`.

One line in the user's rc file, generated fresh from whatever version of `ghostvolumes` is currently installed ŌĆö avoids version skew between a stale copied-and-pasted snippet and the actual binary behavior.

**Non-interactive shells** (CI, `bash -c`, some IDE terminals) still won't source rc files ŌĆö as already noted in ┬¦6, these need `LD_PRELOAD` set globally (e.g. exported from `/etc/environment` or the CI job config) to get even the reactive coverage.

### 8.3 Platform gating

BTRFS ioctls, LD_PRELOAD, and `/proc/self/mountinfo` are all Linux-specific. Gate the crate with `#[cfg(target_os = "linux")]` around the platform-dependent modules, and have `main()` print a clear "GhostVolumes only supports Linux with BTRFS" message and exit cleanly on other platforms, rather than failing to compile or panicking ŌĆö `cargo install` will otherwise happily attempt (and fail confusingly) on macOS/Windows.

### 8.4 Versioning

Standard `cargo install ghostvolumes --force` to upgrade in place; mention `cargo install cargo-update` + `cargo install-update ghostvolumes` in the README as a convenience for keeping it current, since `cargo install` itself has no auto-update mechanism.

### 8.5 Shim debug logging

The shim never prints to stdout/stderr ŌĆö it's injected into arbitrary host processes, and writing to their standard streams risks corrupting a TUI or polluting output the host process doesn't expect. All logging goes to a file instead.

**Two independent knobs, both pure environment variables, resolved once at process start ŌĆö no config file involved:**
- **Debug on/off** ŌĆö `GHOSTVOLUMES_DEBUG` (any non-empty value other than `0` enables it).
- **Log file path** ŌĆö `GHOSTVOLUMES_LOG_FILE`, defaulting to `~/.local/share/ghostvolumes/shim.log` (same data dir as `compiled.tsv`/`preload.so`) if unset.

An earlier draft of this section routed these through a `settings.toml` ŌåÆ compiled `shim.conf` pipeline, mirroring `compiled.tsv`'s "CLI parses TOML, compiles a flat file the shim reads" pattern. Reconsidered: that pattern earns its complexity for `compiled.tsv` because path-matching data is large, structured, and genuinely needs merging across `*.d/` sources. A boolean and a path don't ŌĆö env-var-only removes an entire file format, a hand-rolled parser, a compiled artifact that can go stale relative to its source, and a `reload` side effect, for two settings a shell export already covers cleanly. It also sidesteps a concrete correctness trap the TOML-routed design didn't originally account for: reusing `compiled.tsv` itself for these settings (the "just add another row" instinct) doesn't work, because `Path::starts_with("")` ŌĆö the natural sentinel for a non-path settings row ŌĆö is true for *every* path, so such a row would corrupt `names_for`'s matching for literally every intercepted call rather than just carrying an unrelated setting safely. Pure env vars have no such trap. Persisting a custom log path across sessions just means exporting the env var in the shell rc file already alongside the `LD_PRELOAD` export from `shell-init` ŌĆö the same pattern tools like `RUST_LOG` use.

**Normal mode** logs only critical/important events: a subvolume actually created, or an unexpected error (failed creation, unreadable `compiled.tsv`). **Debug mode** additionally logs every interception decision for every `mkdir`/`mkdirat` call ŌĆö accept (matched, creating) or skip, and specifically *why* skipped (no cache match, already a subvolume, git-tracked) ŌĆö since this is the direct troubleshooting question ("why didn't/did this become a subvolume") the mode exists to answer. This means debug mode logs constantly across every process on the system once `LD_PRELOAD` is set globally ŌĆö expected and documented, not a bug; turn it off once done troubleshooting.

### 8.6 Testing conventions

This project has **one tier** of tests, not the mocked/real split common elsewhere: nearly the whole suite exercises real BTRFS calls directly (raw ioctls, no mocking layer), by deliberate design ŌĆö a test that merely asserts "the mock was called with X" can't tell "our code does X" from "our code is wrong but the mock doesn't know that" apart, which matters even more here than usual since the shim hand-declares its own syscalls with no library layer to lean on for correctness.

Consequence: `cargo test` needs a real BTRFS-backed directory to pass at all. `btrfs_scratch_dir()` (`src/test_support.rs`, duplicated in `tests/shim_ld_preload.rs` ŌĆö a separate integration-test binary with no lib target to import from) resolves one via, in order: the `GHOSTVOLUMES_TEST_BTRFS_DIR` env var if set, else `<CARGO_MANIFEST_DIR>/target/ghostvolumes-test-scratch` (works whenever the checkout itself is on BTRFS). CI (`.github/workflows/ci.yml`) always sets the env var, pointing it at a throwaway loop-mounted image created fresh per run ŌĆö see `ai-work/tasks/ci.plan.md` for the full design. A contributor with local root and `btrfs-progs` can opt in the same way: create their own loop-mounted (or otherwise real) BTRFS filesystem and export `GHOSTVOLUMES_TEST_BTRFS_DIR` before running `cargo test` ŌĆö no code changes needed, same mechanism CI uses.

`tests/btrfs_loopback.rs` is a second, opt-in layer for environments that can create their *own* throwaway filesystem inside the test process tree (via `unshare --user --map-root-user --mount`), rather than depending on one already being mounted somewhere ŌĆö `#[ignore]`d by default (`cargo test -- --ignored` to run), gracefully skipping (not failing) wherever mount privilege isn't available.

`tests/reload_cli.rs`/parts of `tests/shim_ld_preload.rs` invoke the compiled binary as a subprocess with faked `HOME`/`XDG_*` env vars pointing at tempdirs ŌĆö using a deliberately fake, nonexistent `$HOME` value rather than the real invoking user's, so any future accidental fallback to `$HOME`-derived paths fails loudly instead of silently touching the real developer's home.

## 9. Build order

1. TOML config loader with `*.d/` merge logic ŌĆö underpins everything else.
2. Config compiler (`reload`) producing `compiled.tsv`, auto-invoked from `scan --save` / `discover --save`.
3. `scan --save` ŌåÆ `roots.d/00-auto.toml`.
4. LD_PRELOAD shim (single-file, dependency-free Rust) compiled once via `build.rs` at `cargo install` build time (┬¦8.1); `ghostvolumes init` extracts the embedded compiled bytes to disk ŌĆö no `rustc` invocation at runtime.
5. Wire the shim to read `compiled.tsv` (┬¦8.0).
6. `discover` ŌĆö reuses the `scan` inode-256 check, gives an immediate way to dogfood the config format.
7. `shell-init` + cd-hook logic (`ensure` subcommand), gated on project-match rule.
8. `convert`.
9. Platform gating, packaging polish, publish to crates.io.
10. Shim debug logging (┬¦8.5) ŌĆö `GHOSTVOLUMES_DEBUG`/`GHOSTVOLUMES_LOG_FILE` env vars, log-file writes gated by normal/debug verbosity.
11. CI (┬¦8.6, `ai-work/tasks/ci.plan.md`) ŌĆö GitHub Actions: lint, real-BTRFS test matrix, platform-gating (macOS/Windows), real Snapper interop.
12. (Deferred) seccomp-notify supervisor for static binaries, only if real-world need appears.
